SPRINT: Questionario de Satisfacao do Evento (MVP)
==================================================

Contexto do codigo atual (ja existe hoje)
----------------------------------------
- Documentacao/UX do modulo:
  - docs/tela-inicial/menu/Eventos/Novo evento/form_questionario/form_questionario.md
  - referencia visual: playwright-event-detail-questionario.png
- Backend ja tem modelos/migrations (mas nao tem endpoints):
  - app/models/models.py: QuestionarioPagina, QuestionarioPergunta, QuestionarioOpcao (+ tabelas de resposta)
  - alembic/versions/289b4605dc23_init.py: tabelas questionario_* existem
- Backend ainda NAO tem:
  - schemas Pydantic para a estrutura completa do questionario (read/write)
  - endpoints para carregar/salvar configuracao por evento
- Frontend:
  - rota /eventos/:id/questionario existe e aponta para frontend/src/pages/EventQuestionario.tsx (placeholder "Em breve")
  - ainda nao existe integracao com API nem editor (paginas/perguntas/opcoes)


Objetivo do sprint (escopo)
---------------------------
Entregar a tela "Questionario de Satisfacao" (configuracao) por evento:
- editar estrutura (paginas, perguntas, opcoes)
- reordenar paginas/perguntas/opcoes
- salvar e recarregar a estrutura via API
- manter permissao/visibilidade igual ao modulo de eventos (AGENCIA ve apenas eventos da propria agencia)


Fora do escopo deste sprint (vira backlog futuro)
------------------------------------------------
- Fluxo publico do questionario (respondido pelo lead)
- Coleta/persistencia de respostas (questionario_resposta_*)
- Relatorios/metricas de respostas
- Regras de gatilho (ex.: "redireciona para pesquisa" no check-in)


ISSUES (ordem de execucao sugerida; cada uma e a menor entrega possivel)
======================================================================

01) Decisao de contrato: estrutura + API do Questionario (MVP)
-------------------------------------------------------------
Descricao:
- Definir contrato do JSON da estrutura completa (GET/PUT) e alinhar com o modelo atual no DB:
  - pagina: id (read), ordem, titulo (obrigatorio; max 200), descricao (opcional)
  - pergunta: id (read), ordem, texto/enunciado (obrigatorio; max 500), tipo (enum), obrigatoria
  - opcao: id (read), ordem, texto (obrigatorio; max 200), valor_numerico (opcional; fica fora do MVP?)
- Definir como o PUT funciona (MVP):
  - estrategia A: PUT sobrescreve toda a estrutura (delete + insert)
  - estrategia B: PUT faz upsert por id (mantem ids)
- Definir ordenacao/ordem:
  - como `ordem` e calculada (1..N) e validada
  - como tratar itens repetidos/fora de faixa
- Definir regras de validacao (MVP):
  - pagina precisa de titulo
  - pergunta precisa de texto e tipo valido
  - perguntas OBJETIVA_* precisam de pelo menos 1 opcao
  - (decidir) permitir questionario vazio vs exigir >= 1 pergunta para considerar "ativo"
- Definir bloqueios (MVP):
  - (decidir) bloquear PUT/DELETE de pagina/pergunta (via PUT) quando existirem respostas em questionario_resposta_*
  - formato do erro (code + extra.dependencies)
- Definir codigos de erro padrao (ex.: EVENTO_NOT_FOUND, QUESTIONARIO_INVALID_STRUCTURE, QUESTIONARIO_UPDATE_BLOCKED).
Aceite:
- Documento curto no repo (ou update no form_questionario.md) com as decisoes, limites e exemplos de payload.
