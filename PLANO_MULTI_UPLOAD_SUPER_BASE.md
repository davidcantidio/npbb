# Plano Revisado: Multi-Upload Unificado ate a Super Base

## Summary
- Implementar um fluxo de importacao unificado por sessao, nao apenas um batch visual no Bronze.
- O usuario opera uma unica `ImportSession` / `SuperLote` com multiplos arquivos, multiplas linhas de metadados e envio consolidado ate a Super Base.
- Internamente, cada arquivo continua sendo persistido como `LeadBatch`, porque isso encaixa no backend atual e reduz risco.
- O shell `/leads/importar` passa a suportar dois caminhos Bronze:
  - `single`: comportamento atual, preservado
  - `session`: multi-upload, multi-adicao, mapeamento agrupado e promocao consolidada
- O fluxo ETL permanece inalterado.

## Interfaces
- Novas entidades/backend:
  - `ImportSession`
    - `id`
    - `created_by`
    - `status`: `draft | uploading | uploaded | mapping | ready_for_super_base | processing | completed | completed_with_errors | failed`
    - `total_files`
    - `uploaded_files`
    - `mapped_files`
    - `processed_files`
    - `failed_files`
    - `created_at`
    - `updated_at`
  - `LeadBatch.import_session_id`
    - FK opcional para agrupar batches que pertencem a mesma sessao
- Novos contratos publicos:
  - `POST /leads/import-sessions`
    - cria a sessao pai
  - `GET /leads/import-sessions/{id}`
    - retorna status agregado da sessao e resumo dos filhos
  - `GET /leads/import-sessions/{id}/batches`
    - retorna os batches/arquivos filhos
  - `POST /leads/import-sessions/{id}/executar-super-base`
    - orquestra a promocao de todos os batches elegiveis para a Super Base
- Contrato existente ajustado:
  - `POST /leads/batches`
    - adicionar `import_session_id` opcional
    - manter compatibilidade total com o modo atual
- Tipos frontend novos:
  - `BronzeMode = "single" | "session"`
  - `ImportSessionRead`
  - `ImportSessionBatchSummary`
  - `SessionUploadRowDraft` com:
    - `local_id`
    - `file`
    - `file_name`
    - `quem_enviou`
    - `plataforma_origem`
    - `data_envio`
    - `evento_id`
    - `origem_lote`
    - `tipo_lead_proponente`
    - `ativacao_id`
    - `status_ui`
    - `created_batch_id`
    - `mapping_status`
    - `pipeline_status`
    - `error_message`

## Implementation Changes
- Shell de importacao:
  - Em `ImportacaoPage`, separar:
    - fluxo atual `single`
    - fluxo novo `session`
    - fluxo ETL
  - A navegacao canonica passa a aceitar `session_id` para retomar uma sessao:
    - `/leads/importar?step=upload&session_id=<id>`
    - `/leads/importar?step=mapping&session_id=<id>`
    - `/leads/importar?step=pipeline&session_id=<id>`
  - O fluxo legado por `batch_id` continua funcionando para `single`
- Upload multi-arquivo:
  - Criar uma sessao pai antes do envio dos arquivos
  - Cada linha/arquivo vira um `LeadBatch` filho com `import_session_id`
  - Manter a grade editavel com uma linha por arquivo
  - Permitir linhas mistas:
    - `proponente`
    - `ativacao`
  - Reutilizar:
    - `QuickCreateEventoModal`
    - `listReferenciaEventos`
    - `listEventoAtivacoes`
    - `createEventoAtivacao`
    - `updateEvento`
    - `listAgencias`
- Dependencias por linha:
  - Ao selecionar evento:
    - carregar elegibilidade para ativacao
    - carregar ativacoes do evento quando necessario
  - Se o evento nao tiver `agencia_id`:
    - mostrar editor inline de agencia
    - salvar com `updateEvento`
    - atualizar cache local do catalogo
    - desbloquear a linha sem reload
  - Se o evento nao tiver ativacoes:
    - permitir criacao inline da ativacao
- Mapeamento:
  - O mapeamento deixa de ser tratado como navegacao batch isolada por arquivo e passa a ser uma etapa da sessao
  - Cada arquivo continua tendo seu `batch_id` e seu mapeamento real no backend
  - A UI agrupa arquivos por assinatura exata de cabecalho
    - mesma assinatura: permite reaproveitar o mesmo mapeamento
    - assinatura diferente: exige mapeamento separado
  - O endpoint real continua o atual por `batch_id`
  - A experiencia do usuario e consolidada por sessao
- Envio para Super Base:
  - A acao principal da sessao sera "Enviar tudo para Super Base"
  - Backend orquestra os batches filhos elegiveis usando o pipeline Gold atual por `batch_id`
  - Concorrencia limitada no processamento consolidado: `2`
  - Sem rollback global
  - Resultado final agregado por sessao:
    - concluido
    - concluido com erros
    - falhou
- Estado e retomada:
  - A sessao deve ser reidratavel via `GET /leads/import-sessions/{id}` e `GET /.../batches`
  - O usuario pode sair e voltar para a mesma sessao sem perder o contexto
- O que continua igual:
  - modo `single`
  - fluxo ETL
  - contrato atual de mapeamento e pipeline por `batch_id`
  - sem reescrever a Super Base do zero

## Test Plan
- Frontend:
  - selecionar multiplos arquivos cria uma sessao e uma linha por arquivo
  - submit da sessao cria `ImportSession` e chama `createLeadBatch` uma vez por arquivo com `import_session_id`
  - modo `single` continua indo para preview/mapping como hoje
  - ETL segue intacto
  - linha `proponente` envia `tipo_lead_proponente`
  - linha `ativacao` bloqueia sem `ativacao_id`
  - evento sem agencia mostra editor inline, salva e desbloqueia a linha
  - evento sem ativacoes permite criacao inline
  - arquivos com mesmo cabecalho podem reaproveitar mapping
  - arquivos com cabecalhos diferentes exigem mapping separado
  - acao "Enviar tudo para Super Base" dispara processamento consolidado da sessao
  - erros parciais nao apagam sucessos
  - sessao pode ser retomada por `session_id`
- Backend:
  - criar sessao
  - associar `LeadBatch` a sessao
  - ler agregados e filhos da sessao
  - executar processamento consolidado da sessao
  - preservar compatibilidade do `POST /leads/batches` sem `import_session_id`
  - cobrir status agregados da sessao com sucesso parcial e falha parcial

## Assumptions
- "Super Base" corresponde ao estagio final ja alimentado pelo pipeline Gold atual.
- O `LeadBatch` continua sendo a unidade tecnica real de ingestao por arquivo.
- A `ImportSession` vira a unidade de operacao do usuario.
- O v1 nao cria um novo motor de transformacao unificado; ele orquestra de forma consolidada os motores atuais por arquivo.
- O ETL continua fora desse escopo.
