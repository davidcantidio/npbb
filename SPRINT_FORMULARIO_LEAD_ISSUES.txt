SPRINT: Configuração do Formulário de Lead por Evento (MVP)
==========================================================

Contexto do código atual (já existe hoje)
---------------------------------------
- Backend já tem os modelos e migrations:
  - app/models/models.py: FormularioLandingTemplate, FormularioLeadConfig, FormularioLeadCampo
  - alembic/versions/289b4605dc23_init.py cria as tabelas
- Backend ainda NÃO tem:
  - schemas (Pydantic) específicos para esse módulo
  - endpoints para ler/salvar a configuração do formulário de lead por evento
  - endpoint para listar templates/temas (para dropdown do front)
- Frontend ainda NÃO tem:
  - serviços para consumir /evento/{id}/form-config
  - página/aba “Formulário de Lead” no detalhe do evento

Objetivo do sprint (escopo)
---------------------------
Entregar a “Configuração do Formulário de Lead” por evento (painel interno):
- escolher 1 tema/template
- selecionar campos ativos (checkbox) e marcar obrigatórios/opcionais
- visualizar URLs geradas (landing, promotor, questionário, API)
- salvar e recarregar a configuração

Fora do escopo deste sprint (vira backlog futuro)
------------------------------------------------
- Landing page pública funcional capturando leads (POST /lead)
- Fluxo do promotor / check-in
- Questionário público
- Integração Salesforce
- Editor de HTML/CSS do template (no máximo preview)


ISSUES (ordem de execução sugerida; cada uma é a menor entrega possível)
======================================================================

01) Decisão de contrato: “campos ativos” e “obrigatório”
-------------------------------------------------------
Descrição:
- Definir como o backend vai representar campos do formulário.
Decisão proposta (MVP, compatível com o modelo atual):
- “Campo ativo” = existe registro em FormularioLeadCampo.
- “obrigatorio” = bool no registro.
- “ordem” = int no registro.
Aceite:
- Documento curto no PR/README (ou comentário no novo schema) descrevendo a regra.


02) Backend: criar schemas do módulo Formulário de Lead (read/write)
-------------------------------------------------------------------
Descrição:
- Criar arquivo novo: app/schemas/formulario_lead.py
- Incluir modelos Pydantic mínimos:
  - FormularioLandingTemplateRead (id, nome)
  - FormularioLeadCampoRead/Write (nome_campo, obrigatorio, ordem)
  - FormularioLeadConfigRead (evento_id, template_id, urls, campos[])
  - FormularioLeadConfigUpsert (template_id, campos[])
Aceite:
- Imports funcionando e tipagem clara (sem quebrar nada existente).


03) Backend: endpoint listar templates (temas)
--------------------------------------------
Descrição:

Aceite:
- Endpoint responde 200 e ordena por nome.

05) Backend: GET /evento/{id}/form-config (retorna config ou “vazio”)
--------------------------------------------------------------------
Descrição:
- Implementar GET /evento/{id}/form-config
Comportamento MVP:
- Se evento não existir (ou não visível ao usuário), retornar 404/403 conforme regra já aplicada em /evento.
- Se não existir FormularioLeadConfig para o evento:
  - retornar um “config default” (sem persistir) com:
    - evento_id
    - template_id: null
    - campos: []
    - urls: calculadas (ver issue 06) OU vazias por enquanto (ver issue 06)
Aceite:
- Front consegue renderizar um formulário “em branco”.


06) Backend: util para gerar URLs do evento (landing/promotor/questionário/API)
------------------------------------------------------------------------------
Descrição:
- Criar helper (ex.: app/utils/urls.py) para gerar URLs.
- Adicionar env var (ex.: PUBLIC_APP_BASE_URL) com fallback local.
- Definir padrões (MVP):
  - landing:   {PUBLIC_APP_BASE_URL}/landing/eventos/{evento_id}
  - promotor:  {PUBLIC_APP_BASE_URL}/promotor/eventos/{evento_id}
  - quest:     {PUBLIC_APP_BASE_URL}/questionario/eventos/{evento_id}
  - api-doc:   {PUBLIC_APP_BASE_URL}/docs (ou link do Swagger do backend)
Aceite:
- GET /evento/{id}/form-config retorna urls preenchidas (mesmo sem config salva).


07) Backend: PUT /evento/{id}/form-config (criar/atualizar config sem campos)
----------------------------------------------------------------------------
Descrição:
- Implementar PUT /evento/{id}/form-config aceitando:
  - template_id (opcional)
- Upsert:
  - se não existe config, cria
  - se existe, atualiza template_id
Aceite:
- PUT retorna 200 com o config atualizado (template_id persistido).


08) Backend: PUT /evento/{id}/form-config (substituir lista de campos)
----------------------------------------------------------------------
Descrição:
- Estender o PUT para aceitar campos:
  - campos: [{nome_campo, obrigatorio, ordem}]
- Estratégia MVP: “replace all”
  - apaga campos antigos daquela config e recria a lista enviada (em transação)
- Validações mínimas:
  - nome_campo não vazio e max_length <= 80
  - ordem >= 0
  - não permitir nomes duplicados no payload
Aceite:
- Após PUT, GET devolve exatamente a lista enviada.
